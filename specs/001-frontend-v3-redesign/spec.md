## User Scenarios & Testing *(mandatory)*

### User Story 1 — 律师在桌面端开展实时会谈 (Priority: P1)

律师在会谈过程中需要同时关注当事人画像、实时洞察和转写内容。桌面端采用三栏布局：左侧固定展示当事人画像展板，中间主区域滚动展示实时洞察流，右侧可折叠区域展示精简转写参考。

**Why this priority**: 这是产品的核心使用场景，桌面端是律师主要的会谈工作环境，信息架构直接影响律师的决策效率。

**Independent Test**: 律师可以独立进入会谈页面，看到完整的桌面三栏布局，各区域内容正常渲染且互不遮挡。

**Acceptance Scenarios**:

1. **Given** 律师已进入实时会谈页面，**When** 页面加载完成，**Then** 左侧显示当事人画像展板（基本信息、情绪状态、关键主张、风险暴露、已确认事实），中间显示实时洞察流，右侧显示精简转写参考
2. **Given** 律师正在进行会谈，**When** 系统产生新的洞察，**Then** 洞察卡片实时追加到中间区域的顶部或按时间顺序展示
3. **Given** 律师需要更多转写空间，**When** 点击收起右侧转写面板，**Then** 中间洞察区域自动扩展占据释放的空间
4. **Given** 律师需要查看历史内容，**When** 滚动中间或右侧区域，**Then** 仅该区域滚动，其他区域保持固定

---

### User Story 2 — 律师在移动端开展实时会谈 (Priority: P1)

律师在手机或平板等移动设备上参与会谈时，屏幕空间有限，需要以洞察为主、画像和转写为辅的紧凑布局。底部提供 Tab 导航切换三个视图。

**Why this priority**: 移动端适配是律师在外出或庭审间隙使用产品的关键场景，洞察为主的布局确保核心信息不被淹没。

**Independent Test**: 律师可以在移动设备上进入会谈页面，通过底部 Tab 在洞察、画像、转写三个视图间切换，各视图内容完整可读。

**Acceptance Scenarios**:

1. **Given** 律师在移动设备打开会谈页面，**When** 页面加载完成，**Then** 默认展示洞察流视图，顶部显示当事人画像摘要卡片
2. **Given** 律师在移动端查看洞察流，**When** 点击底部「画像」Tab，**Then** 切换至当事人画像完整视图
3. **Given** 律师在移动端查看画像，**When** 点击底部「转写」Tab，**Then** 切换至精简转写参考视图
4. **Given** 律师在移动端使用会谈功能，**When** 系统状态变化（如连接/断开），**Then** 顶部状态指示器实时更新

---

### User Story 3 — 律师在开始会谈前进入入口页 (Priority: P2)

律师首次打开应用或需要开始新会谈时，通过入口页了解产品价值并发起新的加密会话。

**Why this priority**: 入口页是用户旅程的起点，虽然不是会谈中的高频页面，但直接影响用户的第一印象和会话创建流程。

**Independent Test**: 律师可以独立访问入口页，点击「开始新会谈」后成功创建会话并跳转至会谈页面。

**Acceptance Scenarios**:

1. **Given** 律师访问应用首页，**When** 页面加载完成，**Then** 展示产品介绍文案和「开始新会谈」主按钮
2. **Given** 律师在入口页，**When** 点击「开始新会谈」按钮，**Then** 创建新会话并跳转至实时会谈页面
3. **Given** 律师尚未注册声纹，**When** 在入口页看到声纹注册引导链接，**Then** 点击后可跳转至声纹注册流程

---

### User Story 4 — 律师实时接收并处理分析建议 (Priority: P2)

会谈过程中，系统识别出可进一步分析的法律意图时，向律师展示「可分析意图」卡片，律师可选择生成深度分析或忽略。

**Why this priority**: 这是 AI 辅助价值的核心体现，帮助律师在会谈中快速获取深度法律依据和策略建议。

**Independent Test**: 律师可以在会谈中看到待分析意图卡片，点击「生成深度分析」后卡片状态变化，最终展示分析结果。

**Acceptance Scenarios**:

1. **Given** 系统识别到可分析意图，**When** 向律师展示「可分析意图」卡片，**Then** 卡片包含意图描述、置信度/倒计时和操作按钮
2. **Given** 律师看到可分析意图卡片，**When** 点击「生成深度分析」按钮，**Then** 卡片进入运行状态，显示分析进度
3. **Given** 律师看到可分析意图卡片，**When** 点击「忽略」按钮，**Then** 卡片从洞察流中移除或标记为已忽略
4. **Given** 深度分析完成，**When** 结果返回，**Then** 在洞察流中展示对应的法规引用、风险提示或策略建议卡片

---

### Edge Cases

- 当当事人画像数据尚未生成时，画像展板应展示占位状态而非空白
- 当洞察流为空时，中间区域应展示引导文案或空状态提示
- 当转写参考为空时，右侧区域应展示等待转写的提示
- 当网络连接断开时，顶部状态指示器应变为离线状态并提示用户；网络恢复后系统自动重连，并拉取断网期间缺失的历史数据补全到当前视图
- 在移动端横屏状态下，系统强制竖屏展示，提示用户旋转设备
- 当同时出现多个可分析意图时，卡片应按出现顺序堆叠展示，避免信息过载

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须提供入口页面，包含产品介绍、开始新会谈按钮和声纹注册引导
- **FR-002**: 系统必须提供桌面端实时会谈页面，采用三栏布局（画像展板 / 洞察流 / 转写参考）
- **FR-003**: 系统必须提供移动端实时会谈页面，采用单栏布局配合底部 Tab 导航（洞察 / 画像 / 转写）
- **FR-004**: 当事人画像展板必须展示基本信息、情绪状态、关键主张、风险暴露和已确认事实五个模块
- **FR-005**: 实时洞察流必须支持展示多种类型的洞察卡片：可分析意图、法规引用、风险提示、合同条款、行为分析
- **FR-006**: 精简转写参考必须按说话人区分展示会谈文本，支持律/当等角色标识
- **FR-007**: 可分析意图卡片必须包含意图描述、操作按钮（生成深度分析/忽略）和进度/倒计时指示
- **FR-008**: 系统必须提供连接状态指示器，实时反映会谈会话的连接状态
- **FR-009**: 桌面端右侧转写参考面板必须支持折叠/展开，折叠后中间洞察区域自动扩展
- **FR-010**: 移动端底部 Tab 导航必须支持在洞察、画像、转写三个视图间切换
- **FR-011**: 所有页面必须采用深色主题，保持设计稿中定义的色彩体系和视觉层次
- **FR-012**: 移动端会谈页面必须强制竖屏展示，横置设备时提示用户旋转至竖屏

### Key Entities

- **当事人画像（Profile）**: 包含角色、案件类型、会谈轮次等基本信息，情绪状态评分及描述，关键主张列表，风险暴露列表（带等级），已确认事实列表
- **洞察（Insight）**: 包含类型（法规引用/风险提示/合同条款/行为分析）、标题、详细描述、关联法律条文、风险等级（如适用）
- **可分析意图（Suggestion）**: 包含意图描述、预览主题、预览理由、状态（待确认/运行中/已完成）、请求标识
- **转写文本（Transcript）**: 包含说话人角色标识、文本内容、时间顺序
- **会谈会话（Session）**: 包含会话标识、连接状态、录音状态、各实体数据的集合

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 律师可以在 3 秒内从入口页进入实时会谈页面并开始接收数据
- **SC-002**: 桌面端三栏布局在 1440×900 及以上分辨率下各区域内容完整可见，无内容截断或重叠
- **SC-003**: 移动端布局在 390×844 分辨率下各视图内容完整可读，Tab 切换响应时间不超过 200 毫秒
- **SC-004**: 洞察卡片从数据到达至渲染完成的时间不超过 500 毫秒
- **SC-005**: 90% 的律师用户在首次使用桌面端三栏布局时，可以在不阅读说明的情况下找到画像、洞察和转写三个核心区域

## Assumptions

- 后端 API 已提供获取当事人画像、实时洞察、转写文本和会谈历史的数据接口
- 声纹注册流程作为独立功能已在其他规格中定义，入口页仅提供跳转链接
- 深色主题为唯一主题，暂不提供浅色主题切换
- 设计稿中定义的色彩值（如背景色、文字色、强调色）作为最终视觉标准执行
- 桌面端和移动端通过响应式断点自动切换，不单独维护两套代码
- 旧版前端代码将被完全替换，不保留 fallback 或兼容层
- 项目已集成 shadcn/ui（`base-nova` 风格），新组件应优先基于 shadcn 组件扩展，而非从零手写
