# Session Retro（会话回顾）

五阶段 session 回顾。Stop hook 自动触发；也可手动调用 `/session-retro`。

不要对只有一问一答的琐碎 session 调用。

## 核心流程

```
阶段 1：设定基调（Prime Directive 无罪推定）
  → 阶段 2：收集数据（决策时间线 + 五类信号扫描）
  → 阶段 3：生成洞察（AAR 四问 + 5 Whys 追问根因）
  → 阶段 4：决定行动（聚合去重、优先级排序、≤3 条 SMART 行动项）
  → 阶段 5：收尾检查（回顾质量自查）
  → AskUserQuestion 向用户确认
  → 通过 writing-skills 流程归档到分类 skill
```

## 阶段 1：设定基调

应用 Prime Directive（无罪推定）：假设在当时所知、所具备的能力和资源下，每个人都尽了最大努力。目标是找到**系统性缺陷**（流程缺口、知识缺口、环境约束），而非追责。

**红色警报 — 停下来深挖**：如果一个问题可以被"我不小心"解释，说明真根因还没找到。"不小心"背后一定有一个缺失的流程检查点。

## 阶段 2：收集数据

### 2a. 决策时间线

以**决策事件**（而非每条消息）为单位，重构 session：

```
时间顺序：用户说了什么 → 我做了什么决策 → 选了哪个分支 → 结果如何
```

决策事件包括：工具选择时刻、执行顺序选择时刻、方案 A/B 选择时刻、用户打断/纠正时刻、并行 vs 串行选择时刻。

### 2b. 五类信号扫描

沿时间线扫描以下信号：

| # | 信号 | 触发条件 |
|---|------|---------|
| S1 | 用户明确反馈 | "记住这个"/"下次注意"/"又犯了这个错"/"不要这样做" |
| S2 | 重复模式 | 同一类偏离在单 session 中出现 ≥ 2 次 |
| S3 | 决策纠正 | 用户明确否定了我的方向、工具选择或执行顺序 |
| S4 | 更优范式 | 出现了明显更好的做法，且未被现有规则覆盖 |
| S5 | 侥幸/隐患 | 结果对了但推理路径有缺陷；或某个风险在场但恰好没触发 |

**S5 是关键**——传统的"找错误"会漏掉"结果对但路径错"的案例。这些往往更危险，因为它们没有失败反馈信号。

### 2c. 压力/情境因素

记录决策时上下文压力：多任务并行、长对话上下文稀释、用户指令模糊、时间压力。

**阶段 2 产出**：带标签的关键决策时间线。

## 阶段 3：生成洞察

对每个候选信号依次执行：

### 3a. AAR 四问

| # | 问题 |
|---|------|
| Q1 | **预期**发生什么？ |
| Q2 | **实际**发生什么？（决策 → 行动 → 结果的完整链条） |
| Q3 | 什么做**对了**，为什么？（值得固化保留——同等重要） |
| Q4 | 什么可以**改进**，如何改？ |

### 3b. 5 Whys 追问根因

从表面偏离开始，逐层追问"为什么"，直到根因落在以下三个类别之一：

- **流程缺陷**：缺少检查点或强制机制
- **知识缺口**：决策时缺少可用的规则/参考
- **环境约束**：工具/平台的固有限制，需要绕过

**永远不要停在"我不小心"。**那从来不是真正的根因。

### 3c. 因果分类

| 类型 | 定义 |
|------|------|
| **直接原因** | 导致了本次可见问题的因素 |
| **未被触发的隐患**（S5） | 在场但未造成可见后果的风险——往往更有价值 |

**阶段 3 产出**：每条候选的完整分析（AAR 答案 + 5 Whys 链 + 因果/隐患分类）。

## 阶段 4：决定行动

### 4a. 聚合去重

按根因聚合：多个表面不同的信号，如果 5 Whys 追到同一个根因 → 合并为一条。

### 4b. 优先级排序

两维度打分（1–5），乘积排序：

| 维度 | 1 | 3 | 5 |
|------|---|---|----|
| **频率** | 首次出现 | 偶尔出现（每 session 2–3 次） | 几乎每次决策都出现 |
| **影响** | 轻微绕路（< 1 分钟） | 方向错误需重新开始 | 破坏性（数据/文件/安全） |

**频率 ≥ 3 且影响 ≥ 3** 的条目必须此次处理。

### 4c. SMART 格式

| 要素 | 含义 | 检查 |
|------|------|------|
| **S**pecific 具体 | 精确描述什么场景做什么 | 能不能想象出那个时刻？ |
| **M**easurable 可测 | 怎么验证做到了 | 能不能判断是否遵守了？ |
| **A**chievable 可行 | 一条规则就能覆盖 | 一条 skill 条目或 CLAUDE.md 一行够不够？ |
| **R**elevant 相关 | 直接针对根因 | 不是在症状上贴创可贴？ |
| **T**ime-bound 时效 | 下一个 session 生效 | Stop hook 结束之后是否已写入？ |

### 4d. 上限 3 条

**每个 session 最多产出 3 条**行动项。超出的低优先级条目写入 `judge-framework.md` 待观察区，后续 session 再次出现则升级。

**阶段 4 产出**：≤ 3 条 SMART 行动项，每条标注优先级分数。

## 阶段 5：收尾与检查

### 质量自查
- 每条是否追到了根因（而非停在现象）？
- 每条是否 SMART（而非"下次注意"这种无效承诺）？
- 有没有漏掉 S5（侥幸/隐患）信号？
- 检查 `judge-framework.md` 待观察区——之前延迟的条目是否该升级了？

### 回顾回顾本身
- 回顾流程本身有无可改进之处？
- 判断标准是否需要调整？→ 记录到 `judge-framework.md`

### 输出摘要
- 本次 session 时长、决策数、信号数
- 筛除后保留 N 条候选
- 确认后归档 M 条

## 确认

用 AskUserQuestion（多选）向用户展示发现。每条选项显示：信号类型、一句话摘要、目标分类 skill。用户勾选要归档的条目。

## 归档到分类 Skill

对每条确认的条目，用 `category-matcher.md` 确定目标分类。然后：

- **新建分类 skill**：通过 writing-skills 走完整 TDD（RED subagent 基线 → GREEN 最小 skill → REFACTOR）
- **更新已有分类 skill**：跳过 RED（session 观察 = 基线证据），去重比对，追加 + REFACTOR

## 输出格式

每条归档条目使用三段对比结构：

```markdown
### {简短标题}

**总结**：{一句话 — 根因 + 核心原则}

**反例**：
{具体的错误操作 — 命令/代码/行为}
{必须具体到读者能对号入座"我也是这么干的"}

**正确做法**：
{可执行、可验证的步骤或规则}
{必须具体到能照做 — 而非"以后注意"}
```

## 分类 Skill 命名规范

| 层面 | 规范 | 示例 |
|------|------|------|
| 文件夹名 | `sr-{分类}` | `sr-cmd-conventions` |
| Skill name（frontmatter） | `session-retro:{分类}` | `session-retro:cmd-conventions` |

分类 skill 放在 `.claude/skills/` 下，与 session-retro 平级。调用方式：`/session-retro:cmd-conventions`。
