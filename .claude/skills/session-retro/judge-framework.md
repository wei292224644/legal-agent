# 判断标准参考（Judge Framework）

> 在阶段 2–4 中需要详细判断标准时按需加载。

## 信号检测模式

### S1：用户明确反馈

触发词及其含义：

| 触发词 | 含义 |
|--------|------|
| "记住这个"/"remember this" | 高价值模式，通常也是 S4 |
| "下次注意"/"next time" | 需要可执行的纠正 |
| "又犯了这个错"/"wrong again" | 确认 S2 重复模式 |
| "不对"/"不要这样做" | 即刻的 S3 决策纠正 |

### S2：重复模式

按**根因类别**计数，而非表面现象。比如两次不同命令都没 `cd` 就执行——如果根因都是"无 CWD 前置检查"，计为同一种重复模式。

### S3：决策纠正

纠正类别：
- **方向**：用户否定整体方案（如"不要用 subagent 做这个"）
- **工具**：用户指定不同工具（如"用 Edit 别用 Write"）
- **顺序**：用户纠正执行顺序（如"先验证再跑"）
- **范围**：用户收紧/放宽任务范围

### S4：更优范式

识别检查清单：
- [ ] 这个做法产生了明显更好的结果
- [ ] 它**没有**被 CLAUDE.md 或已有 skill 覆盖
- [ ] 它可泛化（不是针对某个文件的临时技巧）
- [ ] 如果 session 开始就知道这个做法，我会做出不同选择

### S5：侥幸/隐患

检测启发式（最难捕捉的信号）：

| 启发式 | 示例 |
|--------|------|
| 命令成功了但 CWD 是碰巧对的 | `pytest` 通过是因为上一条 `cd` 刚好落在对的目录 |
| 并行工具调用成功了但顺序假设是错的 | 两条并行 Bash 调用——其中一条成功只因为另一条还没开始 |
| 用户一个问题阻止了错误路径 | 用户问"你确定吗？"在我执行有缺陷的计划之前 |
| 默认值刚好等于需要的值 | 配置默认值碰巧对，但原因推理是错的 |

**S5 自审提示**："如果环境稍有不同（不同 CWD、不同时序、不同配置），同样的操作会失败吗？"

## 5 Whys 深度检查

### 有效根因类别

| 类别 | 示例 | 有效？ |
|------|------|--------|
| 流程缺陷 | "并行 Bash 调用前没有 CWD 检查机制" | 是 |
| 知识缺口 | "CLAUDE.md 有规则但在决策上下文中不可用" | 是 |
| 环境约束 | "Stop hook 无法获取完整 transcript" | 是 |
| "我忘了" | — | **否 — 继续挖** |
| "我不小心" | — | **否 — 继续挖** |
| "我犯了错" | — | **否 — 继续挖** |

### 5 Whys 反模式

| 反模式 | 修复 |
|--------|------|
| 停在 L2（"我不知道"） | 追问：为什么决策时这个知识不可用？ |
| 循环因果链（A→B→A） | 打破循环：什么外部因素使这个循环持续？ |
| 从 L1 跳到 L5（跳跃式结论） | 每一层必须是真实的因果链接，不能是换个说法 |

## 优先级评分标准

### 频率

| 分数 | 标准 |
|------|------|
| 1 | 所有 session 中首次出现 |
| 2 | 跨 session 第二次出现 |
| 3 | 偶尔出现（每 session 2–3 次） |
| 4 | 经常出现（每 session 4–6 次） |
| 5 | 几乎每个决策点都出现 |

### 影响

| 分数 | 标准 |
|------|------|
| 1 | 轻微绕路（< 1 分钟损失） |
| 2 | 几分钟损失，任务仍完成 |
| 3 | 方向错误需大量返工 |
| 4 | 完全阻塞了一项任务 |
| 5 | 破坏性（数据丢失、文件损坏、安全问题） |

## 待观察列表格式

在 `judge-framework.md` 中维护待处理区：

```markdown
## 待观察条目

| # | 条目 | 首次出现 | 频率 | 影响 | 已延迟 session 数 |
|---|------|---------|------|------|------------------|
| 1 | {一句话描述} | 2026-05-20 | 2 | 2 | 1 |
| 2 | {一句话描述} | 2026-05-20 | 1 | 4 | 1 |
```

**升级规则**：如果任意待观察条目的频率 ≥ 3 或影响 ≥ 5，升级为当前 session 必处理项。
