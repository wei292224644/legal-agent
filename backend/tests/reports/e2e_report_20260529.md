## 四、关键发现与建议

### 4.1 性能瓶颈

| 组件 | 平均延迟 | 瓶颈原因 |
|------|---------|---------|
| IntentRouter | 1404ms |  instructor + Pydantic 结构化输出，temperature=0.1 |
| ProfileAgent | 455ms | 短 prompt、低 max_tokens，无结构化约束 |
| HeavyAgent | 7957ms | 长 system prompt + skills + tool 调用 + 长输出 |

**建议**: HeavyAgent 是主要延迟来源，考虑：
1. 对 `analyze_quick` 启用流式输出（streaming）减少首 token 等待
2. 缓存 skills 加载（已实现 `_load_skills` lru_cache）
3. 复杂分析考虑异步预生成（在 pending 期间后台预跑）

### 4.2 准确率风险点

1. **IntentRouter 对"事实陈述"过度敏感**: 第 10、16 行 client 的应答性陈述被判为 `record_only`（simple），而人工标注为 `ignore`。这会导致不必要的系统响应打断律师节奏。
   - **建议**: 增加"对话上下文中的被动应答"识别规则，降低 simple 判定阈值。

2. **complex vs simple 边界模糊**: 第 23 行"需要准备哪些证据？"被判 simple，但期望 complex。
   - **建议**: 补充训练样本或调整 prompt 中"需要深度分析"的定义。

3. **抖动行集中在客户开场陈述**: 第 2 行"被公司违法解除了"在 `risk_evaluation` 和 `query_law` 之间抖动。
   - **建议**: 这是 temperature=0.1 下的正常 LLM 波动，可考虑降到 0 或增加 few-shot 示例。

### 4.3 ProfileAgent 覆盖率

- 73% 空返回率**符合预期**，但平均每轮仅 0.40 条画像提取，在信息密度高的剧本（如 theft 27 条 / 61 轮 = 0.44/轮）仍有提升空间。
- **建议**: 检查滑动窗口（n=6）是否足够覆盖跨轮事实关联，或增加 co-reference 解析。

### 4.4 HeavyAgent 异常短输出

- 30 次 `analyze_quick` 中约 50% 返回 4 字符，疑似模型返回空内容或 stale 检查触发。
- **建议**: 增加 `analyze_quick` 的日志打点，区分"stale 取消"和"模型空返回"。
